# ADR-004 — Due-compass windows and the morning-done marker

- **Status:** Accepted
- **Date:** 2026-06-12
- **Deciders:** Claude (build agent), per specs/README §3 (least data, fastest return)

*(Numbering note: ADR-002 and ADR-003 are reserved by spec references — Circles
transport encryption and payments respectively — and will be written when those
decisions are due.)*

## Context

THR-01 says the Threshold shows "the due Compass (morning or evening)" but does not
define *due*, nor what happens after a compass is answered. Two gaps:

1. **Windows.** When is each compass offered?
2. **Doneness.** The evening compass leaves a Reflection row (THR-03), so "done
   today" is already derivable. The morning compass deliberately stores nothing —
   the answer is an orientation, not a record — so without some marker the morning
   door stays open all day after it was already walked through, inviting a second
   visit the product never asked for.

## Decision

### Windows

- Morning compass: due from `morning_hour` until `evening_hour`.
- Evening compass: due from `evening_hour` until midnight; it takes over even if
  the morning was missed (the day is nearly over — reflection serves better than
  a stale orientation question).
- Before `morning_hour`: nothing is due; the Threshold is one quiet line.

### Morning-done marker

A single nullable `morning_done_date` column on the profile holds the local date
key of the last completed morning compass, overwritten each time. Marked when the
person accepts Today's Opening (the "i will" step). Yesterday's value simply
expires by comparison with today; it is never read for anything else.

What we deliberately do **not** keep:

- No history of completed mornings — one overwritten value, so no run of
  consecutive days is even computable from stored data (INV-2 by construction).
- No timestamps, no count, no skipped-day record.
- The morning answer itself remains unstored (least data).

Evening doneness is derived from the existing Reflection row for today's date;
no second marker is introduced.

## Covenant Test (INV-1..9 walk-through)

| Invariant | Effect |
|---|---|
| INV-1 (no feed) | Neutral; the Threshold still shows ≤ 3 elements. |
| INV-2 (no engagement mechanics) | Strengthened: an answered compass disappears for the day, removing any pull to re-open the app; the single overwritten date makes consecutive-day mechanics impossible to build on stored data. |
| INV-3 (app closes itself) | Unchanged; flows still end in terminal screens. |
| INV-4 (two notifications) | Untouched; the marker never schedules anything. |
| INV-5 (on-device data) | One date string in the encrypted local DB. |
| INV-6 (one-tap erasure) | Lives in the profile row, already wiped by `eraseAll()`. |
| INV-7 (no shame copy) | A missed morning shows nothing — no "you missed it" state exists. |
| INV-8 (identical app) | Neutral. |
| INV-9 (safety first) | Neutral. |

## Consequences

- After both compasses are answered, the Threshold is just the quiet line and the
  unplug door — the app visibly has nothing more to offer, which is the point.
- The due slot is computed on app open and on flow exit; an app left open across
  an hour boundary shows the stale slot until the next interaction. Accepted: no
  timers run in the background to refresh it (nothing watches the clock to call
  the person back).
