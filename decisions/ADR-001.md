# ADR-001 — Stack, state management, persistence, and content direction

- **Status:** Accepted
- **Date:** 2026-06-10
- **Deciders:** Ivan Marianek (product owner), Claude (build agent)

## Context

`02-architecture.md` recommends Flutter or React Native and delegates the final
choice, the state-management pattern, and the persistence implementation to an ADR.
The build is done by an AI agent inside a sandboxed Linux container; the owner builds
and runs on a Windows host. Launch targets are iOS 16+ and Android 10+,
feature-identical, offline-first, with home/lock-screen widgets as "the product's
true front door".

## Decision

### Stack: React Native + TypeScript (bare, not Expo)

- Single codebase for both platforms; TypeScript end-to-end including the scoring
  engine (`core/scoring`), which stays a pure, UI-free module.
- **Bare** workflow (community CLI), not Expo managed: widgets, SQLCipher, Screen
  Time / UsageStats access all require native modules; Expo's managed layer adds
  indirection without removing that work.
- Native modules we will write or configure ourselves: home/lock-screen widgets
  (WidgetKit / Glance), screen-time signals (DeviceActivity / UsageStatsManager),
  OS focus-mode integration for the Quiet.

### State management: Redux Toolkit

The spec requires a unidirectional pattern. Redux Toolkit is the RN-native default,
serializable by design (helps the route-graph and erasure tests), and its action log
makes the anti-engagement invariants auditable: product logic consuming forbidden
inputs (session length, open counts — NFR-A5) is greppable at the action level.

### Persistence: op-sqlite + SQLCipher; key in Keychain/Keystore

- `@op-engineering/op-sqlite` compiled with SQLCipher (`"sqlcipher": true` in
  package.json `op-sqlite` config) — encrypted at rest (NFR-P2).
- The database key is generated on first run (32 random bytes) and stored via
  `react-native-keychain` (iOS Keychain / Android Keystore). `eraseAll()` deletes
  the key as well as the data (INV-6, NFR-P4).
- Migrations: plain ordered SQL files applied by a tiny versioned runner in
  `core/storage` — no ORM; the data model in `03-data-model.md` is small and stable.

### Testing

- Jest + React Native Testing Library; `core/*` modules are pure TS with 100%
  coverage required for `core/scoring` (NFR-R1).
- Covenant rails run in CI before anything else (see `ci/` and NFR-A1..A5, NFR-P3).

## Content direction (documented deviation from the spec canon)

The spec pack and source documents describe a cross-traditional reading canon
(scripture among it). **Product owner decision (2026-06-10): Inward avoids religious
texts and religious framing entirely — it is for all people with good heart.**
Library readings, Paths, intake copy and dares draw on secular, literary,
philosophical and nature voices (public domain where possible). The product's
poetic vocabulary ("spirit over matter", "soul") remains; doctrine, scripture and
prayer framing do not. The two documents in `docs/previous/` are explicitly out of
scope as sources.

## Covenant Test (INV-1..9 walk-through)

| Invariant | Effect of this decision |
|---|---|
| INV-1/2/3 (no feed, no engagement, app closes) | Neutral; enforced by rails in CI regardless of stack. RN navigation graph is statically analyzable for the TerminalScreen rule (NFR-A3). |
| INV-4 (two notifications) | Local-only via a 2-slot scheduler module; no push infra in v1. |
| INV-5 (on-device data) | op-sqlite+SQLCipher keeps all personal data local and encrypted; no analytics SDKs — dependency lint enforces (NFR-P3). |
| INV-6 (one-tap erasure) | Key deletion + table wipe implementable atomically in `core/storage`. |
| INV-7 (no shame copy) | Content direction above; copy review checklist applies. |
| INV-8 (identical app) | No code path reads payment state for feature gating; integration test planned per 04 §6. |
| INV-9 (safety first) | Unaffected by stack; on-device classifier feasible in RN via a small native/JS model (decided in M2). |

## Consequences

- Native widget work is scheduled explicitly (M1 for the one-line widget, M3 for
  Craving Button widgets) rather than assumed free.
- The sandbox dev image gains JDK 17 + Android SDK so unit tests, lint, covenant
  gates and Android debug builds run inside; iOS builds happen on a Mac later
  (store submission is an M4 concern).
- Expo tooling (EAS, OTA JS updates) is forgone; content OTA is our own signed
  bundle pipeline per `02-architecture.md §Content` and is independent of app code.
