# ADR-002 — Circles transport: E2E ciphertext relay

- **Status:** Proposed
- **Date:** 2026-06-12
- **Deciders:** Ivan Marianek (product owner) — pending; drafted by Claude (build agent)

## Context

Circles (CIR-01..02) are groups of 5–8 people sharing exactly one weekly
written reflection and one shared dare. `03-data-model.md` leaves the v1
transport open: an end-to-end encrypted relay, or invite-link + manual share
(no server at all). Constraints: only the weekly reflection written *for the
circle* ever leaves a device, E2E-encrypted (INV-5); the server, if any,
stores ciphertext only (CIR-02); threat model T3 must hold (a compromised
relay can delay, never read).

## Options

### A — No server: invite link + manual share

The app composes the weekly reflection as an encrypted blob the person
shares through any channel they already trust (a messaging app, even paper).
- For: zero infrastructure, zero new endpoints (NFR-P1 list unchanged),
  strongest possible privacy posture.
- Against: depends on a third-party channel exactly where we promise to need
  none; weekly rhythm becomes manual labor; joining/loss flows are brittle;
  the shared dare needs a round-trip the channel cannot give.

### B — Minimal E2E ciphertext relay (proposed)

A dumb mailbox keyed by circle id: members post and fetch ciphertext;
keys are generated on-device and exchanged through the invite link itself
(the link IS the key material, shown once, never sent to the relay).
- For: the weekly rhythm works offline-then-sync; the relay learns only
  ciphertext sizes and timings; fits NFR-P1's allowance "(e) optional Circle
  ciphertext relay"; one new endpoint, fully described in the threat model.
- Against: real infrastructure to run; traffic metadata exists (accepted in
  threat model T3 residual risk); key recovery is impossible by design —
  losing the link means rejoining (acceptable: a circle is people who can
  re-invite you).

## Decision (proposed)

Option B. Crypto: libsodium sealed boxes with a per-circle symmetric key
derived from the invite secret; the relay never sees a user identifier —
circle id is a random opaque token. Message retention: 30 days, then
deleted; the journal copy on each device is the durable record.

## Covenant test

- INV-1/2: words only, no feeds, no counts of any kind — the relay has no
  read receipts, no presence, no member activity signals. PASS.
- INV-5: ciphertext only off-device; keys never leave members. PASS.
- INV-6: leaving a circle deletes the local key; eraseAll covers it. PASS.

## Consequences

M4 implements the relay client behind a port (like storage and the
companion gateway); the relay service is a separate, public-source repo.
This ADR awaits the owner's acceptance before any code.
